
# 12. Analyzer Rules

Date: 2026-05-04

## Status

Accepted

## Context

To ensure code quality, consistency, and maintainability across the CQRS ecosystem, we are adopting custom Roslyn analyzers. These analyzers will enforce best practices, detect anti-patterns, and provide actionable feedback to developers at compile time. For effective tracking, communication, and management of analyzer diagnostics, we need a standardized format for error and warning codes generated by our analyzers.

## Decision

We will use a structured code format for all Roslyn analyzer diagnostics:

**Format:** `YAMC[EXT][NNN]`

- `YAMC`: Static prefix for all CQRS analyzer rules.
- `[EXT]`: Optional 2–4 letter extension identifier for the relevant module or technology (e.g., `SBKA` for ServiceBus Kafka, `EFPG` for EntityFramework Postgres). If not applicable, this section is omitted.
- `[NNN]`: Three-digit, zero-padded incremental rule number (e.g., `001`, `002`).

**Examples:**
- `YAMC001`: Core rule, applies to all projects.
- `YAMCSBKA001`: ServiceBus Kafka-specific rule.
- `YAMCEFPG002`: EntityFramework Postgres-specific rule.

**Rationale:**
- The format is human-readable, easy to search, and scalable as new modules and rules are added.
- The extension segment allows for clear ownership and filtering of rules by technology or concern.

## Implementation Guidelines

- All new analyzer rules must use the standardized code format.
- Rule codes must be unique across the entire codebase.
- The extension segment should be registered/documented for each new module.
- Analyzer documentation must include the rule code, description, rationale, and examples of compliant and non-compliant code.
- Rule numbers should be assigned sequentially within each extension/module.

## Consequences

### Positives

- **Consistency:** Developers can quickly identify the source and purpose of a diagnostic.
- **Scalability:** New rules and modules can be added without ambiguity or code conflicts.
- **Traceability:** Issues can be referenced unambiguously in documentation, code reviews, and bug reports.
- **Tooling:** Enables filtering and suppression of diagnostics by code prefix or extension.

### Negatives

- **Overhead:** Requires discipline to maintain unique codes and update documentation.
- **Learning Curve:** New contributors must learn the code format and extension registry.

## Future Considerations

- Automate rule code assignment and validation to prevent duplicates.
- Integrate analyzer rule documentation with the developer portal or CI pipeline.
- Periodically review and deprecate obsolete rules to keep the analyzer set relevant.
